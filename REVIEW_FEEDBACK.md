# 🌪️ Vortex POS — تقرير المشاكل والتحسينات المطلوبة (Claude Review)

> **الغرض من هذا المستند:** تسليم واضح لفريق التطوير يوضح ما هو ناقص، ما يحتاج تعديل، وما هو مقترح لرفع جودة المنتج وجاهزيته للبيع.

---

## 1. مشاكل أمنية — أولوية عالية 🔴

### 1.1 غياب Rate Limiting على الـ API
- **المشكلة:** لا يوجد أي حد لعدد الطلبات على endpoints الخاصة بالـ Authentication أو أي route أخرى.
- **الخطر:** النظام مكشوف لهجمات brute-force على تسجيل الدخول، وإمكانية استنزاف الخادم (DoS).
- **المطلوب:** تطبيق `express-rate-limit` أو ما يعادله على جميع routes، وبصفة خاصة:
  - `POST /auth/login` → حد أقصى 10 محاولات / دقيقة لكل IP.
  - جميع routes الحساسة (الأرصدة، الفواتير) → حد أقصى معقول.

---

### 1.2 غياب Audit Log (سجل التدقيق)
- **المشكلة:** لا يوجد أي تسجيل لمن قام بأي عملية ومتى، خاصة على البيانات المالية.
- **الخطر:** في حالة نزاع أو اختلاس، لا يمكن إثبات أي شيء. يُعدّ هذا deal-breaker لأي عميل enterprise.
- **المطلوب:** جدول `audit_logs` يُسجّل تلقائياً:
  - اسم المستخدم (user_id).
  - نوع العملية (CREATE / UPDATE / DELETE).
  - الجدول والـ record المتأثر.
  - القيمة القديمة والجديدة (JSON).
  - الوقت الدقيق والـ IP.

---

### 1.3 JWT — لا يوجد آلية إلغاء (Token Revocation)
- **المشكلة:** عند طرد موظف أو تغيير صلاحياته، الـ token القديم يظل صالحاً حتى انتهاء مدته.
- **المطلوب:** إضافة جدول `token_blacklist` أو استخدام Refresh Token مع قصر مدة Access Token (15 دقيقة).

---

## 2. مشاكل في الـ RBAC (نظام الصلاحيات) — أولوية عالية 🔴

### 2.1 الأدوار المتاحة غير كافية
- **الوضع الحالي:** دورين فقط — `manager` و `cashier`.
- **المشكلة:** لا تغطي متطلبات أغلب العملاء الجادين.
- **المطلوب إضافته:**

| الدور | الصلاحيات |
|---|---|
| `owner` | عرض فقط — تقارير، أرباح، إحصائيات (بدون تعديل) |
| `supervisor` | كل صلاحيات الكاشير + قدرة على إلغاء الطلبات والخصومات |
| `accountant` | الوصول للدفتر المالي والتقارير دون الأوردرات |
| `cashier` | الوضع الحالي — كما هو |

---

### 2.2 الـ Middleware لا يتحقق من صلاحيات دقيقة (Fine-grained)
- **المشكلة:** التحقق الحالي على مستوى "هل هو manager أم لا" فقط، بدون تحكم في actions محددة.
- **المطلوب:** نظام permission-based حيث كل action لها permission منفصلة (مثلاً: `can_delete_order`، `can_view_reports`، `can_edit_price`).

---

## 3. غياب Multi-tenancy — أولوية متوسطة للـ SaaS 🟡

- **المشكلة:** النظام مبني لعميل واحد (single-tenant). لا يمكن حالياً تشغيله كـ SaaS لأكثر من عميل على نفس الـ instance.
- **الحل المؤقت (المقترح الآن):** deploy منفصل + database منفصلة لكل عميل على Supabase (سهل وسريع).
- **الحل الدائم (مستقبلي):** إضافة عمود `tenant_id` على جميع الجداول الرئيسية وعزل البيانات على مستوى الـ query. يستلزم refactor كبير.

---

## 4. ثغرات في الـ API والـ Validation — أولوية متوسطة 🟡

### 4.1 لا يوجد Global Error Handler
- **المشكلة:** أخطاء غير متوقعة قد تُعيد stack trace أو رسائل داخلية للـ client.
- **المطلوب:** middleware مركزي للأخطاء يُعيد رسائل موحدة وآمنة، ويسجّل التفاصيل داخلياً فقط.

### 4.2 لا يوجد Input Sanitization واضح
- **المشكلة:** Joi يتحقق من الشكل لكن لا يوجد ذكر لـ sanitization ضد XSS أو SQL injection على inputs الحرة (مثل أسماء المنتجات، الملاحظات).
- **المطلوب:** تطبيق `dompurify` أو `sanitize-html` على أي حقل نصي حر، حتى مع Sequelize ORM.

### 4.3 File Upload غير مذكور (إن وُجد)
- إذا كان النظام يسمح برفع صور للمنتجات، يجب التحقق من النوع والحجم وتخزينها خارج الـ web root.

---

## 5. نقاط ضعف في الـ Frontend — أولوية منخفضة 🟢

### 5.1 لا يوجد ذكر لـ CSRF Protection
- الـ JWT في الـ header يحمي نسبياً، لكن إذا كان مخزناً في Cookie يجب إضافة CSRF tokens.

### 5.2 Session Timeout على الـ UI
- لا يوجد ذكر لـ auto-logout بعد فترة خمول على الواجهة، وهو مطلب أمني أساسي في بيئات البيع.
- **المطلوب:** Idle timeout بعد 15–30 دقيقة مع تنبيه للمستخدم.

### 5.3 Error States و Loading States غير موثقة
- لا يوجد ذكر لكيفية تعامل الـ UI مع حالات الشبكة البطيئة أو انقطاع الإنترنت، وهو سيناريو شائع في بيئات الـ POS.
- **المقترح:** Offline queue للطلبات مع sync عند عودة الاتصال (PWA approach).

## 7. ملاحظات على البنية التقنية

- **SQLite + PostgreSQL دعم مزدوج:** جيد للتطوير، لكن يجب توثيق الفروق في السلوك بين الاثنين خاصة في الـ locking.
- **Sequelize ORM:** مناسب، لكن يُفضّل مراجعة الـ N+1 query problem في صفحات العرض الكبيرة.
- **Auto-Backup كل 12 ساعة:** جيد، لكن يجب التأكد من وجود backup على server خارجي.
- **الـ Stress Test على 50 أوردر متزامن:** ممتاز، يُنصح بتوثيق نتائجه كـ benchmark رسمي.
